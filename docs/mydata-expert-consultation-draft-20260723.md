# Rentme 마이데이터 전문가 상시자문 내부 상담 초안

작성일: 2026-07-23  
용도: [DOW-553](/DOW/issues/DOW-553) 내부 검토 산출물  
상태: 보드 승인 전 초안 — 온라인 접수·개인정보 입력·파일 업로드·이메일 발송 금지

## 1. 신청 개요

- 지원사업: `2026년 마이데이터 전문가 상시자문`
- 주관/수행: 과학기술정보통신부 / 한국데이터산업진흥원
- 지원대상: 마이데이터 사업화를 추진하는 기업 및 개인 예비창업자
- 지원분야: 법·제도, 기술, 사업화 분야별 원포인트 자문
- 신청기간: 2026-05-11 ~ 2026-12-18
- 권장 상담 제목: `Rentme 동의 기반 임대 확인 리포트의 수집·가공·제공·철회 구조 사전 검토`
- 권장 우선분야: 1순위 `법·제도`, 2순위 `기술`, 3순위 `사업화`

이 문서는 상담 신청 전 내부 정리안이며 법률 의견서가 아니다. 실제 서비스 정책, 동의 문구 및 제3자 제공 구조는 전문가 검토 후 확정한다.

## 2. 서비스와 상담 목적

Rentme(입주해)는 세입자가 임대차에 필요한 확인 항목을 정리하고, 임대인이 세입자의 명시적 동의 범위 안에서 제한된 정보를 열람하도록 돕는 주거 임대 매칭 서비스다. 현재 MVP는 세입자 입력 프로필, 재직·소득·신용 관련 확인 상태, 이전 임대인 레퍼런스, 원본 증빙에서 분리한 검증 상태, 서비스 내부 참고용 요약 신호를 다룬다.

이번 상담의 목적은 다음 네 가지다.

1. 정보주체가 직접 입력하거나 제출한 정보와 외부 확인·레퍼런스에서 파생된 정보의 법적 성격을 구분한다.
2. 수집·가공·제공 단계별 동의와 고지, 목적 제한, 보유기간, 철회·정정 절차를 확정한다.
3. 원본 증빙 대신 최소한의 확인 상태만 제공하는 기술 구조가 적절한지 검토한다.
4. 개인신용평가 또는 임차인 자동 선별로 오인되지 않는 제품·사업화 경계를 정한다.

## 3. 데이터 수집·가공·제공 흐름 1페이지 요약

```text
[1. 정보주체 입력/제출]
세입자 본인
  ├─ 프로필·거주 선호·생활 조건 직접 입력
  ├─ 재직·소득·신용 관련 확인용 증빙 제출
  └─ 이전 임대인에게 레퍼런스 요청
            │
            ▼
[2. 목적별 저장 및 검증]
  ├─ 원본 증빙: 접근이 제한된 evidence/document 영역
  ├─ 추출·검토: OCR 또는 운영 검토 상태
  ├─ 검증값: 확인 여부·상태·갱신 시점 중심의 파생값
  └─ 레퍼런스: 응답, 정정·이의제기 상태를 별도 관리
            │
            ▼
[3. 내부 가공]
  ├─ 프로필 완성·확인 상태·레퍼런스 등을 요약 신호로 변환
  ├─ 원본 서류와 자유서술 원문은 기본 제공 대상에서 분리
  └─ 전체 점수 산식은 운영 내부 정보로 유지
            │
            ▼
[4. 정보주체의 제공 동의]
  ├─ 열람자: 특정 사용자 또는 역할
  ├─ 목적: 임대 후보 검토 등 명시된 목적
  ├─ 범위: 허용 필드와 연락처 공개 여부
  ├─ 기간: 시작일·종료일
  └─ 통제: 철회, 정정·이의제기
            │
            ▼
[5. 임대인 제한 열람]
  ├─ 동의·목적·필드·유효기간을 확인한 뒤 허용 정보만 제공
  ├─ 연락처는 별도 허용 시에만 공개
  ├─ 원본 증빙과 내부 전체 산식은 제공하지 않음
  └─ 허용/거절 결과와 열람 목적을 접근 로그로 기록
            │
            ▼
[6. 사후 통제]
  ├─ 동의 철회 후 신규 접근 차단
  ├─ 검증값·레퍼런스에 대한 정정/이의제기
  ├─ 유효기간 만료 및 재검증
  └─ 보유기간 종료 시 삭제 또는 비식별·통계 처리 정책 적용
```

### 단계별 정보 범위

| 단계 | 다루는 정보 | 외부 제공 원칙 | 현재 확인된 제품 통제 |
| --- | --- | --- | --- |
| 직접 입력 | 연령대, 가족 유형, 희망 지역, 반려동물, 흡연 여부, 거주 조건, 자기소개 | 사용자가 선택한 항목만 | 프로필과 열람 필드 분리 |
| 증빙 제출 | 재직·소득·신용 관련 문서 | 원본은 기본 제공 금지 | 원본 증빙과 파생 검증값 분리 |
| 검증·가공 | 확인 여부, 검토 상태, 갱신 시점, 요약 신호 | 목적에 필요한 최소 상태만 | `allowed_fields`, `allowed_purposes` 적용 가능 |
| 레퍼런스 | 이전 임대인 응답, 추천 여부, 정정·이의제기 상태 | 원문보다 요약 상태 우선 | 레퍼런스 분쟁·정정 요청 별도 관리 |
| 제한 열람 | 특정 임대인에게 허용된 리포트 필드 | 기간·목적·상대방 단위 최소 제공 | `valid_until`, `revoked_at`, 연락처 별도 허용, 접근 로그 |

## 4. 전문가에게 요청할 핵심 질문 5개

### 질문 1 — 법·제도: 단계별 동의와 제3자 제공의 단위

세입자가 직접 입력한 프로필, 재직·소득·신용 관련 증빙, 파생된 확인 상태, 이전 임대인 레퍼런스를 수집·가공하고 특정 임대인에게 제한적으로 보여줄 때 각각 어떤 법적 근거와 동의가 필요한가?

- 회원가입·서비스 제공 필수 동의와 임대인 열람 동의를 분리해야 하는가?
- `재직`, `소득`, `신용 관련 확인`, `레퍼런스`, `연락처`를 선택 동의로 세분해야 하는가?
- 특정 임대인·특정 매물·특정 목적·기간 단위 동의가 적절한가?
- 제3자 제공, 처리위탁, 개인정보의 단순 열람 중 어떤 구조로 분류되는가?

### 질문 2 — 법·제도: 신용정보·자동화된 평가·차별 오인 경계

확인 상태와 레퍼런스를 내부 요약 신호에 반영하고 임대인이 후보 탐색에 참고하도록 하는 구조가 개인신용평가, 신용정보업 또는 자동화된 임차인 선별로 해석될 수 있는 경계는 무엇인가?

- `신용 인증`, `신뢰점수`, `검증된 세입자` 대신 어떤 표현과 고지가 적절한가?
- 숫자 점수나 최저점 필터를 외부에 노출하지 않고 `확인 완료/추가 확인 필요` 수준만 제공하면 위험이 의미 있게 줄어드는가?
- 설명, 정정, 이의제기, 재산정 및 사람의 최종 검토 절차를 어느 수준으로 갖춰야 하는가?
- 가족 유형, 반려동물, 흡연 여부 등 생활 조건을 검색·정렬에 쓰는 경우 부당 배제 위험을 어떻게 제한해야 하는가?

### 질문 3 — 기술: 최소수집·원본 분리·파생값 보관 설계

원본 증빙을 제한 저장하고 임대인에게는 확인 여부·검토 상태·유효기간 등 파생값만 제공하는 설계가 개인정보 최소처리 원칙에 부합하려면 어떤 추가 통제가 필요한가?

- 원본, OCR 추출값, 검증값, 접근 로그별 적정 보유기간과 삭제 트리거는 무엇인가?
- 암호화, 키 분리, 역할 기반 접근통제, 관리자 열람 승인, 다운로드 차단 중 필수 또는 권고 통제는 무엇인가?
- OCR/AI 처리에 외부 API를 사용할 때 위탁·국외 이전·학습 이용 금지 고지와 계약 조항은 어떻게 구성해야 하는가?
- 철회 시 원본과 파생값, 백업, 이미 생성된 요약 신호를 각각 어떻게 처리해야 하는가?

### 질문 4 — 기술/법·제도: 동의 철회·정정·접근 로그

현재 구상한 `열람자 + 목적 + 허용 필드 + 연락처 여부 + 유효기간 + 철회` 통제와 접근 로그가 충분한가?

- 철회 효력 발생 시점과 이미 열람·저장한 임대인에게 필요한 고지 또는 재이용 제한은 무엇인가?
- 정보주체가 누가 언제 어떤 목적으로 무엇을 열람했는지 확인할 수 있어야 하는가?
- 레퍼런스 원문과 파생 검증값에 대한 정정·이의제기 중에는 외부 제공을 중단하거나 `검토 중`으로 표시해야 하는가?
- 감사 목적 접근 로그의 적정 항목과 보유기간은 무엇인가?

### 질문 5 — 사업화: 안전한 MVP와 확장 경계

예비창업 단계의 MVP가 마이데이터형 서비스로 사업화 가능성을 검증하면서도 규제 위험을 낮추려면 어떤 기능 범위로 시작하는 것이 적절한가?

- 1단계는 사용자 직접 입력과 본인 열람, 2단계는 특정 임대인 1회성 공유, 3단계는 외부 데이터 연계로 확장하는 순서가 적절한가?
- 외부 기관 API 연계 전 준비해야 할 동의, 계약, 보안, 데이터 품질 및 책임분담 요건은 무엇인가?
- 유료화 대상을 데이터 판매가 아닌 본인 리포트 관리·동의 기반 공유·B2B workflow로 두는 것이 적절한가?
- 전문가 추가 검토가 필요한 시점과 관련 인허가·신고 가능성을 판정하는 체크포인트는 무엇인가?

## 5. 개인정보 없는 mock 첨부 목록

모든 mock은 `가상 데이터` 워터마크를 넣고 실제 사용자·운영 식별자를 사용하지 않는다.

| 번호 | mock 첨부 | 포함 내용 | 제외 내용 |
| --- | --- | --- | --- |
| 1 | `서비스 흐름도 1장` | 입력 → 검증 → 동의 → 제한 열람 → 철회 | 실제 시스템 구성도, endpoint, secret |
| 2 | `가상 세입자 본인 리포트 1장` | 가명, 가상 연령대, 확인 상태, 유효기간, 정정 버튼 | 실제 회사명·소득·신용등급·서류 이미지 |
| 3 | `제공 동의 화면 1장` | 가상 임대인, 목적, 필드 체크박스, 기간, 연락처 별도 선택 | 실제 이름·전화·이메일·주소 |
| 4 | `임대인 제한 열람 화면 1장` | 마스킹 이름, 허용된 확인 상태, 만료 안내 | 원본 증빙, 상세 사유, 전체 점수·산식 |
| 5 | `철회·접근내역 화면 1장` | 가상 열람자, 가상 시각, 목적, 철회 상태 | 실제 IP, user ID, 운영 로그 |
| 6 | `데이터 항목표 1장` | 항목군, 수집 출처, 목적, 제공 여부, 보유기간 초안 | DB schema, table/column 전체 목록 |

첨부 권장 순서는 1 → 3 → 4이며, 신청 화면에서 첨부가 필수가 아니면 먼저 본문만 제출하고 전문가 요청 시 승인된 mock만 추가한다.

## 6. 외부 제출 제외 기준

다음 자료는 이번 자문에 제출하지 않는다.

- 실제 사용자 이름, 연락처, 이메일, 주소, 생년월일, 회사명, 소득, 신용등급·점수
- 실제 이전 임대인 정보, 레퍼런스 자유서술 원문, 분쟁·정정 사유 원문
- 실제 재직·소득·신용 관련 원본 서류, OCR 원문, 이미지·PDF
- 운영 DB, DB dump, 실제 table/column 전체 명세, 운영 access log, IP, user ID
- `API key`, `.env`, DB connection string, webhook secret, 인증 token, 외부 서비스 credential
- `lib/trust-score.ts` 원본 코드, 전체 점수 산식, 범주별 세부 배점, 최대값, `trust.score_breakdown`
- 운영 아키텍처의 취약점을 드러낼 수 있는 내부 endpoint, 네트워크·인프라 구성, 보안 설정 원문
- 개인정보가 포함된 화면 캡처, 로그, 이메일, 계약서, 사업자·대표자 원본 서류

## 7. 상담 신청 본문 초안

```text
Rentme(입주해)는 세입자가 임대차에 필요한 확인 항목을 정리하고, 특정 임대인이 세입자의 명시적 동의 범위 안에서 제한된 정보를 열람하도록 돕는 주거 임대 매칭 서비스입니다.

현재 MVP에서는 사용자가 직접 입력한 프로필, 재직·소득·신용 관련 확인 상태, 이전 임대인 레퍼런스, 원본 증빙에서 분리한 검증 상태를 다룹니다. 원본 증빙은 제한 저장하고, 외부 열람에는 확인 여부·유효기간 등 최소한의 상태값만 제공하는 방향을 검토하고 있습니다. 열람자, 목적, 허용 항목, 연락처 공개 여부, 유효기간, 철회 상태를 기준으로 접근을 통제하고 접근 이력을 남기는 구조입니다.

다음 사항을 자문받고 싶습니다.

1. 직접 입력 정보, 증빙, 파생 검증값, 레퍼런스를 수집·가공하고 특정 임대인에게 제공할 때 필요한 단계별 동의와 제3자 제공 구조
```

## 8. 보드 승인용 Go/No-Go 체크리스트

### 필수 확인값

- [ ] 신청 주체를 `개인 예비창업자` 또는 `기업` 중 하나로 확정했다.
- [ ] 대표자명·기업명·사업자 상태를 신청 계정 정보와 일치하도록 확인했다.
- [ ] 외부 상담용 이메일과 전화번호 사용을 승인했다.
- [ ] 상담 제목과 본문, 질문 5개를 검토했다.
- [ ] 상담 분야 우선순위를 `법·제도 → 기술 → 사업화`로 승인했다.
- [ ] mock 첨부 여부와 허용할 첨부 번호를 승인했다.
- [ ] 제출 파일 전체에서 실제 개인정보·운영 식별자·credential을 재검수했다.
- [ ] 전체 점수 산식·세부 배점·`trust.score_breakdown`이 포함되지 않았음을 확인했다.
- [ ] 온라인 접수 버튼을 누를 담당자와 접수 계정을 지정했다.

### 판정 기준

- `Go`: 모든 필수 확인값 완료, 제출본문 승인, 개인정보 없는 첨부만 승인, 최종 파일 재검수 완료.
- `Conditional Go`: 본문은 승인됐지만 신청 주체·연락처·첨부 범위 중 하나가 미확정. 미확정값 입력 전 접수 금지.
- `No-Go`: 실제 개인정보·원본 증빙·운영 DB·secret·전체 산식 제출이 요구되거나, 신청 주체 자격을 확인할 수 없음.

### 현재 판정

`Conditional Go` — 내부 상담 초안은 준비됐으나 신청 주체, 대표자/기업명, 외부 연락처, 접수 계정, mock 첨부 범위에 대한 보드 승인이 필요하다.
